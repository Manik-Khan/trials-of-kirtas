- **Never change a theme var to fix a per-page problem** — hardcode page-specific overrides on the page itself. Reuse existing theme vars for *new* UI.
- **Any Supabase table created via raw SQL needs an explicit `GRANT`.** House style is `grant ... to authenticated` (an older note said `anon, authenticated` — the live tables only grant `authenticated`; match the live tables).
- **Realtime checklist:** table in the `supabase_realtime` publication + socket authenticated via `setAuth` (re-applied on token rotation) + `replica identity full`.
- **Provision Supabase `profiles` by auth `uid`, never by assuming email.** A seed keyed on `email` silently no-ops if run before that user's first login (the `auth.users` row doesn't exist yet for OTP). `role` (privileges) and `character_key` (which PC) are **independent** — staff/Narrator correctly has `character_key = null`. There is no `narrator` role; `dm` *displays* as "Narrator". (See the Auth/profiles playbook below.) **Backfilling/repairing by email-join to `auth.users` ordered by `last_sign_in_at desc` is duplicate-safe** (picks the active login) and needs no access to the user's machine.
- **A write policy granted `to authenticated` gates on "logged in," not "is a member."** With open OTP that's anyone → gate party/character WRITES on `public.is_member()` (overseer/dm/player), not bare `authenticated`.
- **A SECURITY DEFINER RPC invoked with the SERVICE key has no `auth.uid()`** (service role isn't a user), so `is_overseer()`-style self-gates inside it fail. Call overseer-gated RPCs from the BROWSER with the user's session token; reserve the service key for reads the client can't do (e.g. listing `auth.users`).
- **"nothing to commit" is a red flag** that a file never actually changed, not a success.
- Don't reverse a correct earlier refusal/decision under pressure; judge cumulative output.

## Architecture (locked)

- **Three-role model:** `overseer` / `dm` / `player`, enforced via Supabase RLS + a column-guard trigger (`combatants_guard_columns`) and `public.is_staff()`.
- **`combatants` table is truth during combat**, with `source_key` binding to character data. Column guard pins staff-only fields from players (owner, name, side, hidden, max_hp, **disposition**, **in_combat**); players may change hp, conditions, x, y, **initiative**.
- **`combat.html`** is the dedicated live battle-map page.
- **`battle.js`** is the cross-page HUD; it's backend-agnostic and shares state through a **pluggable backend seam** (`window.__battle.useBackend({...})`). `combat.html` hands it a Supabase `combatants`-backed backend (`load`, `save`, `saveConditions`, `setInitiative`, `advanceTurn`, `logRoll`, `subscribe`). The turn/round bridge (`setTurn`/`advanceTurn`/`pushTurnToHud`/`activeSourceKey`) is on `main`. **Option A (done): `combat.html` AND `party.html` now install ONE shared backend — `combatants-backend.js` (`window.makeCombatantsBackend(deps)`) — via `useBackend`; write logic lives once, each page injects its own glue (cache/channel/logger).** The turn engine is the **`advance_turn()` `SECURITY DEFINER` function** both surfaces call via `rpc('advance_turn',{p_encounter})` — computes next from the FULL roster (RLS-safe), server-side off-turn guard, logs the single masked replay turn event.
- **One active encounter at a time** (`encounters`, uuid PK).
- **`public.characters` table is the live source for character-sheet data** (`structural` / `vitals` / `inventory` / `equipment` / `currency` / `bio` / `notes`), same RLS + column-guard pattern as `combatants` (`characters_guard_columns` pins identity cols `key`/`owner`/`created_at`; everything else owner-editable). Git `data/characters/<key>.json` stays the export backstop. **`party.html`** is the immersive character-sheet takeover (portrait wall → in-page game menu), reading live via **`character-data.js`** (`CharacterData.loadParty` / `loadCharacter` / `canEdit`, on `window.__tok.sb`).
- **Right dock on `combat.html` (UX-AUDIT-1/1b):** one collapsible panel with **icon-over-label tabs** (Feed · Display · Bestiary · Combat · Scenes; staff tabs absent for players) — `buildDock`/`dockRegister`/`dockOpen`, `dockRegister` takes an optional short `tab` label ('Game feed' → 'Feed'). Eye (view-as-player) + minimize ✕ live in the tab-row corner. Collapsed = a slim ❮ right-edge handle with an unread dot; the handle reopens to the last tab. A **roll ticker** (latest roll, plain-text-stripped body, click → feed) shows whenever the Feed pane isn't visible; it seeds from the newest archived roll and inherits hidden-roll masking (RLS + an `effectiveStaff()` check for the eye preview). Tab row is overflow-proof (`flex:1 1 0; min-width:0` + label ellipsis; corner `flex:none; margin-left:auto`) — a long pane title once pushed ✕ off-screen; don't regress this. Statblock slide-out now anchors `right:0` (the rail is gone) and KEEPS its visibility gate (its closed `translateX(100%)` state parks exactly where the handle + ticker live and must not eat clicks — that bug shipped once). Zoom bar at `right:12px`. Clicking the active tab does NOT collapse (deliberate change from the rail's toggle; ✕ is the close).
- **Roll-modifier seam (battle.js, public):** `RS` state (adv/dis/bless/guidance) drives all HUD rolls; `window.__battle.toggleRS(key)` / `getRS()` / `onRSChange(cb)` let a page host its own toggle surface. On `combat.html` the feed's mods row IS that surface and the roller popup is hidden (page-specific CSS); every other page keeps the roller.
- **Cross-page roll logging:** `feed-bridge.js` (loaded after `battle.js` on all 8 non-combat pages) sets `window.__battle.onLogRoll`, the fallback battle.js uses when the backend lacks `logRoll`. It posts HUD rolls to the feed with session + active-encounter stamps (30s-TTL context cache). `combat.html` never loads it; its full backend wins regardless.
